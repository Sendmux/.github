# Contributing

Thanks for taking the time to improve Sendmux projects.

## Before you start

- Read the repository README and linked docs first.
- Search existing issues and pull requests.
- For bugs, use the bug report template and include a minimal reproduction.
- For documentation issues, link the affected page or section.
- For security issues, follow [SECURITY.md](https://github.com/Sendmux/.github/blob/main/SECURITY.md) instead of opening a public issue.

## Issues

Good issues include:

- A clear summary.
- The affected product surface, package, tool, or docs page.
- Expected behaviour and actual behaviour.
- Steps to reproduce where applicable.
- Sanitised logs or screenshots where useful.

Do not paste API keys, passwords, tokens, webhook secrets, customer data, private account details, or full production payloads into GitHub.

## Pull requests

Pull requests should:

- Stay focused on one change.
- Explain the user-visible effect.
- Include tests or verification evidence where the repository supports it.
- Update docs when behaviour changes.
- Avoid unrelated formatting or refactors.

Maintainers may ask for changes, split a large pull request, or close stale work that cannot be reviewed safely.

## Documentation

Public docs should be accurate to shipped behaviour, use AU/British spelling, and avoid exposing private implementation details. If the source behaviour is unclear, open an issue instead of guessing.
